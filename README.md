# Garmin Coach Lab

Garmin Coach Lab is a local, rule-based endurance coaching prototype that combines wearable activity data with short-term life context to generate explainable weekly training guidance.

The project started as an exploration of Garmin activity data, but evolved into a layered coaching engine:

```text
Garmin activity data
→ Metrics layer
→ Rule-based decision engine
→ Manual life context
→ Coach Context
→ Markdown report
→ LLM narrator
```

The key idea is simple:

> Garmin knows what I did.  
> It does not know what my current life situation allows.  
> A useful coach needs both.

## Why this project exists

Wearable devices provide useful training data such as distance, duration, heart rate, and performance estimates. However, they do not fully understand real-life constraints such as:

- travel or vacation
- family responsibilities
- poor sleep
- work stress
- equipment availability
- active pain or injury notes
- whether running, cycling, or indoor training is actually possible that week

Garmin Coach Lab adds a small but important missing layer: **manual context intake**.

This allows the system to make more realistic and safer weekly recommendations.

For example:

```text
Garmin data:
- 3 runs in the last 7 days
- 0 cycling sessions
- load ratio is elevated

Default rule:
- Add an easy Z2 cycling session

Manual context:
- User is on vacation
- Bike/trainer is not available
- Running is available

Final decision:
- Do not recommend cycling this week
- Maintain easy running rhythm
- Avoid increasing running volume
```

## Current status

This is a local MVP.

It can currently:

- fetch Garmin activity summaries
- fetch Garmin race predictor signals
- calculate weekly and monthly training metrics
- compare recent load against baseline
- classify progression status
- apply manual life/training context
- produce an explainable `coach_context.json`
- render a technical weekly Markdown report
- generate a controlled LLM prompt
- generate a user-facing coach message via OpenAI API
- perform preflight checks for local dependencies and environment variables
- validate decision behavior with scenario matrix tests

## Architecture

### 1. Raw / Source Layer

Garmin activity and performance data are pulled through Python scripts.

Main outputs:

```text
data/activity_summary.json
data/performance_summary.json
```

### 2. Metrics Layer

The metrics layer summarizes recent training load.

Example metrics:

```text
- last 7 days total hours
- last 30 days total hours
- weekly running sessions
- weekly cycling sessions
- average heart rate
- weekly baseline
- load ratio
```

### 3. Rule Engine

The rule engine makes deterministic decisions.

It does not use an LLM.

Example progression statuses:

```text
restart
below_baseline
stable
productive_build
caution_growth
sharp_rebuild_low_absolute_load
spike_risk
```

The rule engine decides things such as:

- Should weekly load increase, decrease, or stay the same?
- Are intervals allowed?
- Should running volume be maintained?
- Should cycling be prioritized?
- Is the current training load risky?

### 4. Manual Context Layer

The user provides current life and training context before generating a coach message.

Example:

```json
{
  "context_period": "current_week",
  "family_status": "normal",
  "sleep_disrupted": false,
  "workload": "low",
  "travel": true,
  "training_environment": "vacation",
  "bike_available": false,
  "trainer_available": false,
  "running_available": true,
  "injury_notes": null,
  "energy_level": "normal",
  "available_days": ["monday", "wednesday", "friday"],
  "user_note": "I am on vacation. I can run, but I do not have access to my bike or trainer."
}
```

This context can modify the final recommendation.

Important distinction:

```text
Travel alone does not automatically mean recovery.
Equipment availability affects what is actually recommended.
Low energy, poor sleep, family pressure, or active injury notes can soften the plan.
```

### 5. Coach Context

The main serving-layer artifact is:

```text
data/coach_context.json
```

It combines:

- athlete profile
- metrics
- performance signals
- rules
- manual context
- final decision
- metadata

This file is the single source of truth for renderers and LLM narration.

### 6. Renderers

There are two main outputs:

```text
data/weekly_review.md
data/coach_message.md
```

`weekly_review.md` is a technical/debug-style report.

`coach_message.md` is the user-facing coach message generated by an LLM.

The LLM does not decide the plan. It only explains the decision already made by the rule engine.

## Setup

This project is designed to run locally with Python.

### 1. Create and activate a virtual environment

On Windows:

```cmd
python -m venv .venv
.\.venv\Scripts\activate
```

### 2. Install dependencies

```cmd
python -m pip install -r requirements.txt
```

### 3. Configure environment variables

Copy the example environment file:

```cmd
copy .env.example .env
```

Then open `.env` and set your OpenAI API key:

```text
OPENAI_API_KEY=your_openai_api_key_here
```

The `.env` file is intentionally ignored by Git.

### 4. Create your local athlete profile

```cmd
copy athlete_profile.example.json athlete_profile.json
```

Edit `athlete_profile.json` with your own profile, goals, constraints, weekly targets, and available equipment.

This file is ignored by Git because it may contain personal information.

### 5. Enter current manual context

Before generating a coaching decision, enter the current weekly context:

```cmd
python update_manual_context.py
```

This creates:

```text
data/manual_context.json
```

Manual context captures information that Garmin does not know, such as travel, sleep disruption, family load, available equipment, training environment, energy level, and active pain notes.

### 6. Run the pipeline

Full pipeline:

```cmd
python run_pipeline.py
```

This refreshes Garmin data, rebuilds coach context, renders reports, generates the controlled LLM prompt, and generates the user-facing coach message.

Run without refreshing Garmin data:

```cmd
python run_pipeline.py --skip-garmin
```

Run without calling the LLM API:

```cmd
python run_pipeline.py --skip-llm
```

Run fully offline from existing local data:

```cmd
python run_pipeline.py --skip-garmin --skip-llm
```

### 7. Run scenario tests

The scenario matrix validates that the decision engine behaves correctly across different life contexts.

```cmd
python run_scenario_matrix.py
```

Example scenarios include:

- normal home week
- vacation without bike or trainer
- child sick / disrupted sleep week
- active injury note
- no running availability

The test summary is generated locally under:

```text
data/scenario_matrix_summary.md
```

## Pipeline preflight checks

`run_pipeline.py` performs a lightweight preflight check before running the pipeline.

It checks:

```text
- active Python executable
- working directory
- .env loading status
- garminconnect availability, unless Garmin refresh is skipped
- openai availability, unless LLM generation is skipped
- OPENAI_API_KEY availability, unless LLM generation is skipped
```

This makes setup problems easier to diagnose, especially when switching between local repositories or virtual environments.

## OpenAI API key

The LLM step requires an OpenAI API key.

Recommended local setup:

```cmd
copy .env.example .env
```

Then edit `.env`:

```text
OPENAI_API_KEY=your_openai_api_key_here
```

Temporary CMD session setup is also possible:

```cmd
set OPENAI_API_KEY=your_api_key_here
```

Do not commit API keys to Git.

## Design principles

### The LLM does not make training decisions

The system follows this rule:

```text
Rules decide.
LLM explains.
```

The LLM receives a controlled prompt based on `coach_context.json`.

It is instructed not to add:

- new intense workouts
- intervals
- tempo runs
- long runs
- unsupported medical advice

### Context matters

The same Garmin data can lead to different decisions depending on life context.

Example:

```text
Same activity data
+ normal week
→ maintain load and add easy Z2 cycling

Same activity data
+ vacation + no bike/trainer
→ maintain easy running, do not recommend cycling

Same activity data
+ poor sleep + child sick
→ reduce or maintain load, recovery priority
```

### Renderers do not compute business logic

Markdown and LLM prompt renderers only consume prepared data.

They do not calculate metrics or change decisions.
